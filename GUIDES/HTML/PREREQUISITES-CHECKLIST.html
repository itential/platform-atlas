Before you start — who are you?
This is embedded in your exported JSON file so your Itential contact can identify your responses.
Required
Workstation
Why we need this

Atlas is a CLI tool that runs on your machine — not on your servers. Your workstation is the control plane for every audit.

What it unlocks

A supported Python version and pip allow Atlas to install and run correctly. The .whl file is Atlas itself. A secure credential backend means Atlas can store sensitive secrets — OAuth tokens, database URIs, SSH passphrases — without ever writing them to a plain-text config file.

Without it

Atlas cannot be installed or launched. Without a credential backend, secrets would need to be re-entered every session and could not be stored securely between runs.

Required
Itential Automation Platform (IAP)
Why we need this

IAP is the primary system being audited. Atlas authenticates using OAuth2 — the Client ID and Secret let it obtain short-lived access tokens to call IAP's REST API. This is strictly a read-only connection: Atlas never modifies IAP configuration or state.

What it unlocks

Atlas can collect the IAP platform version, installed application inventory, active services, and runtime configuration directly from the live system. This data forms the backbone of the compliance report and is required for the majority of ruleset evaluations.

Without it

The IAP audit cannot run at all. Every IAP-related rule — the large majority of the ruleset — will be unevaluable. Without IAP data, the compliance report is not meaningful.

Required for full audit
MongoDB
Why we need this

MongoDB is IAP's primary persistent data store. Atlas connects using the pymongo driver to run read-only diagnostic commands — the same ones a DBA would use to inspect server health. No application data or IAP-owned collection contents are ever read.

What it unlocks

Atlas can validate MongoDB's authentication mode, TLS configuration, journaling, oplog size, replica set health, and storage-layer settings that directly affect IAP's data integrity and availability under load.

Without it

All MongoDB-related rules will show as SKIP in the report. Misconfigurations such as disabled authentication or an undersized oplog won't surface — you lose all visibility into storage-layer compliance.

SSH Access — MongoDB Nodes
ℹ️ MongoDB auditing can be skipped if not needed — leave the URI blank during Atlas setup and MongoDB-related rules will show as SKIP in the report.
Required for full audit
Redis
Why we need this

Redis is IAP's real-time message broker, task queue, and session cache. Atlas connects via redis-py to read server configuration using CONFIG GET — it does not read application data, keys, or values stored by IAP.

What it unlocks

Atlas can validate Redis settings including maxmemory policy, persistence mode (AOF/RDB), bind address, TLS configuration, and ACL setup — all settings that affect IAP's job scheduling, event handling, and session reliability.

Without it

All Redis-related rules will show as SKIP. Redis misconfigurations that could cause IAP job failures, session drops, or data loss under memory pressure won't surface in the report.

SSH Access — Redis Nodes
ℹ️ Like MongoDB, Redis auditing can be skipped by leaving the URI blank during setup. Redis rules will appear as SKIP in the report.
Required
Deployment Topology
Why we need this

Before Atlas can connect to anything, it needs a map of your environment. Topology defines which hosts run which services, how they relate to each other, and what deployment mode your installation uses. Without this map, Atlas doesn't know where to send connections.

What it unlocks

Atlas builds its entire capture plan from the topology — which hosts to SSH into for each role, which addresses to connect to for MongoDB and Redis, and which nodes are primary vs. secondary. This drives the full audit workflow from initial capture through to report generation.

Without it

Atlas cannot run a capture. Topology is not optional infrastructure detail — it is fundamental configuration Atlas requires before doing any work. An incorrect topology means Atlas will connect to the wrong hosts or miss entire services entirely.

Optional
Automation Gateway 4
Why we need this

Automation Gateway 4 is the REST API-based gateway service for IAP integrations. If it's present in your deployment, it has its own configuration that Atlas audits separately from IAP core. Atlas queries the Gateway 4 API directly because automation-gateway.db is the authoritative configuration source — the properties.yml file on disk can be stale after first boot.

What it unlocks

Atlas can capture Gateway 4's runtime configuration, installed version, enabled integrations, and connectivity settings — validating them against the same ruleset that governs the rest of your deployment. API-sourced data is more reliable than reading config files directly.

Without it

If you have Gateway 4 deployed but skip this section, all Gateway 4 rules will show as SKIP and your report will have a gap in gateway coverage. If you genuinely don't use Gateway 4, checking the skip toggle below has no effect on the rest of the audit.

ℹ️ Gateway 4 uses a REST API as its primary data source. Atlas reads automation-gateway.db via GET /config — the properties.yml file on disk may be stale after first boot and is only used as a fallback if the API is unreachable.
Optional
Automation Gateway 5
Why we need this

Automation Gateway 5 is the newer Python-based gateway. Unlike Gateway 4, it has no REST API — all of its configuration is read via SSH by inspecting the virtual environment and reading environment variables from the service unit. Atlas uses the same SSH connection already established for the core audit.

What it unlocks

Atlas can report the installed Gateway 5 package version and validate its environment configuration, ensuring the settings meet compatibility requirements for your IAP version. No additional ports or credentials are required beyond the SSH setup covered in the Platform, MongoDB, and Redis sections.

Without it

If you have Gateway 5 deployed but skip this section, all Gateway 5 rules will show as SKIP and your report will have a gap in gateway coverage. The SSH user's read permissions simply need to cover the Gateway 5 virtual environment directory — no new network access is needed.

ℹ️ Gateway 5 is entirely SSH-based — there is no REST API to query. Atlas reads package version information and environment variables over the same SSH connection used for the rest of the audit. No additional network ports need to be opened beyond SSH.

All applicable prerequisites are confirmed. Run pip install platform_atlas-1.6-py3-none-any.whl on your workstation to get started, then follow the Installation & Usage Guide to configure your first environment and run your first audit.
