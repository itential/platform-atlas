Platform Atlas — User Guide

Installation & Usage Guide

Everything you need to get from a fresh install to a full compliance report — step by step, with no prior Platform Atlas experience required.

Installation

Up and running in minutes

Platform Atlas ships as a Python wheel (.whl) file. You need Python 3.11 or newer — no other dependencies to manage. Install it with pip, then verify it's working with the version command. Atlas creates its config and session data automatically in ~/.atlas/ on first run.

$ pip install platform_atlas-1.6-py3-none-any.whl
Collecting platform-atlas...
Installing collected packages: platform-atlas
Successfully installed platform-atlas-1.6
$ platform-atlas --version
platform-atlas 1.6 · Python 3.11
First-Time Setup

One wizard, everything configured

The first time you run platform-atlas it detects no configuration and launches an interactive setup wizard. You can also run it any time with platform-atlas config init.

The wizard has two phases. Phase 1 captures global settings — your organization name and theme. Phase 2 creates your first environment: credentials for IAP, MongoDB, and Redis (stored in your OS keyring, never in plain text), and your deployment topology (standalone, HA2, or custom) including SSH access details for each server.

When the wizard finishes, you're ready to run your first preflight check.

Sessions

One session, full context

In Platform Atlas, a session is the central unit of work. When you create a session, you bind three things to it: the environment to audit, the ruleset (the set of validation rules), and the profile (a deployment-specific overlay like 'standalone' or 'ha2-with-gateway').

Those bindings are locked in. Switch to a different session and all three switch automatically — no more forgetting to change the ruleset when you switch environments. Sessions also track your progress through the pipeline: capture, validate, report.

  • Create a session with platform-atlas session create
  • Switch with session switch
  • Edit bindings (before capture) with session edit
Running an Audit

Four commands. One complete audit.

Once a session is created, auditing your deployment is a four-step pipeline. Capture connects to every server and service in your deployment and collects configuration data — IAP API responses, MongoDB server status, Redis CONFIG, Gateway runtime data, SSH config files, and log samples.

Validate evaluates that data against every rule in your ruleset, producing PASS, FAIL, SKIP, or ERROR results for each. Report generates an interactive HTML page and opens it in your browser.

Or run all three at once: platform-atlas session run all

🔍

Preflight checks

Before your first audit, run platform-atlas preflight to test connectivity to every service — IAP API, MongoDB, Redis, SSH targets, and config files. It tells you exactly what failed and why.

📋

Rulesets and profiles

A ruleset is the set of best-practice rules Atlas validates against. A profile is a deployment-specific overlay (standalone, HA2, HA2 with Gateway). Both are bound to your session at creation.

🌐

Multiple environments

Create an environment for each deployment — dev, staging, production. Each has its own credentials, topology, and organization name. Sessions tie to a specific environment, so switching is always clean.

⚡

Headless and automated

For cron jobs or CI pipelines, use --headless to suppress all interactive prompts. Combine with --import-dir to batch-import pre-collected files without any interaction.

Session Management

Switch, compare, and share

As you build up sessions over time, Atlas gives you tools to manage them. session list shows all sessions with their environment, status, and progress.

session switch restores the full context of any previous session — environment, ruleset, and profile all switch with it. session diff compares two sessions and generates an HTML comparison report highlighting rules that improved or regressed. session export packages the report and metadata into a ZIP for sharing.
