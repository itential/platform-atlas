Platform Atlas — User Guide

Review Compliance Report for Actionable Insights

Every number, status, and finding explained — plus practical examples showing exactly how to fix what the report surfaces.

Report Layout

Four sections, one complete picture

The Platform Atlas HTML report is self-contained — open it in any browser, no internet required. It has four main sections working top-to-bottom.

The compliance score gives you the headline number and a per-category breakdown. The results table lists every rule evaluated in this session with its status and message. Extended validation surfaces pattern-based findings that single-value rules can't catch — things like log error rate trends and adapter version consistency. Log analysis shows the most frequent error patterns from Platform application logs during the capture window.

Compliance Score

The score tells you where you stand

The compliance score at the top of the report is the percentage of evaluated rules that passed. Rules marked SKIP — which happens when data couldn't be collected for a service — are excluded from both the numerator and denominator. So a 90% score on 80 evaluated rules means the same thing as a 90% score on 60 rules: 90% of what Atlas could check, passed.

The score is broken down by category — Platform, MongoDB, Redis, Gateway4, and Gateway5 — so you can immediately see which area needs the most attention. Target 90% or above on all categories for a healthy production deployment.

Rule Statuses

Four outcomes, clear meaning

Every rule in the results table has one of four statuses. PASS means the rule evaluated and the configuration met the expected value — nothing to do. FAIL means the rule evaluated but the configuration does not meet the expected value — this needs attention, and the message in the report explains exactly what was found and why it matters.

SKIP means data wasn't available to evaluate the rule — this is common when a service was unreachable during capture or when a rule doesn't apply to this deployment type. SKIP is not the same as PASS. ERROR means the rule evaluation itself failed, usually due to a type mismatch or version difference — report these to your Itential contact.

Severity Levels

Not all failures are equal

Every rule has a severity level that tells you how urgently a FAIL needs attention. Critical failures directly threaten data integrity, availability, or security — fix these before your next business cycle. Examples include an unhealthy MongoDB replica set, an incorrect Redis eviction policy, or an unsupported software version.

Warning failures represent configuration that deviates from best practice in ways that may cause problems under load or over time — address these in your next planned maintenance window. Info failures are observations with low immediate risk: default accounts still active, audit retention at default values, optional features not configured. Address these before compliance reviews.

Remediation

Read the message, apply the fix

Every FAIL in the report includes a plain-English message explaining what was found and why it matters. Here are three real examples with their fix steps.

RDS-002 Redis MaxMemory Policy Critical

“Redis maxmemory-policy is not set to noeviction — with other policies, Redis may silently evict important data when memory pressure increases.”

  1. Open the Redis configuration file at /etc/redis/redis.conf.
  2. Locate or add the maxmemory-policy directive.
  3. Set it to: maxmemory-policy noeviction
  4. Restart Redis to apply: sudo systemctl restart redis
  5. Verify the running value with: redis-cli CONFIG GET maxmemory-policy
MDB-002 MongoDB Bind IP Warning

“MongoDB is bound to 0.0.0.0 (all interfaces) — this exposes the database to all network traffic and is a security risk in production.”

  1. Open the MongoDB configuration file at /etc/mongod.conf.
  2. Under the net: section, replace the wildcard with specific IPs: bindIp: 127.0.0.1,<platform-server-ip>
  3. Only include addresses required for Platform connectivity and local administration.
  4. Restart MongoDB to apply: sudo systemctl restart mongod
  5. Verify the binding with: ss -tlnp | grep 27017
IAG-003 Gateway HTTP Server Threads Critical

“HTTP server threads are below the recommended value of 3× the CPU core count — this can create a bottleneck under load and cause request timeouts.”

  1. Open the Automation Gateway properties.yml file.
  2. Calculate 3× the number of CPU cores on the server (e.g. 16 cores → 48 threads).
  3. Set http_server_threads to that value.
  4. Save the file and restart Automation Gateway: sudo systemctl restart automation-gateway

After fixing, re-run platform-atlas session run all to confirm the rules now pass.

Best Practices
Getting the most from your report
🔺

Prioritize by impact

Start with Critical FAILs that affect current availability (unhealthy replica sets, services down). Then address security (open bindings, default accounts). Then configuration correctness (thread counts, eviction policies). Info-level items last.

📤

Share the report

The HTML report is self-contained — email it, drop it in a shared folder, or attach it to a ticket. To include session metadata, use platform-atlas session export which creates a redacted ZIP package.

📊

Compare across audits

After making fixes, create a new session and run the full audit again. Then use platform-atlas session diff baseline-session follow-up-session to generate an HTML comparison showing exactly what improved, what regressed, and what stayed the same.

📝

Export in other formats

Need the results in a spreadsheet or data pipeline? Use --format csv or --format json when generating the report. These exports include the environment name and all rule metadata alongside each result.
